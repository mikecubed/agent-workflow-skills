# Implementation Plan: Workflow Skill Throughput and Coordination Improvements

**Date**: 2026-04-02 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `.sdd/workflow-throughput-9f2k4m1q/spec.md`

## Summary

Improve the existing three-skill workflow architecture (`parallel-implementation-loop`, `pr-review-resolution-loop`, `final-pr-readiness-gate`) by adding bounded discovery/scout behavior, progress-based coordinator rescue policies, shared factual briefs, durable artifact requirements, convergence/disagreement rules, and lightweight outcome measures. All changes target the existing skill SKILL.md files, shared documentation, artifact templates, model configuration, and structural tests. Persistent teams, squads, fleet-style long-lived orchestration, and new separate orchestration skills are **explicitly out of scope**.

## Technical Context

**Language/Version**: Markdown (skill definitions), JavaScript/Node 22+ (tests, scripts)
**Primary Dependencies**: Node.js built-in test runner (`node --test`), `npm pack`
**Storage**: Git-committed Markdown files and YAML config; no database
**Testing**: `npm test` (structural layout validation), `npm run validate:runtime` (CLI integration)
**Target Platform**: GitHub Copilot CLI and Claude Code plugin runtimes
**Project Type**: Shared plugin repository (skill definitions as Markdown protocols)
**Performance Goals**: Reduce repeated discovery across roles; reduce premature subagent abandonment; preserve review quality
**Constraints**: Skills must remain repo-agnostic and reusable; plugin manifests must stay synchronized; three-skill split preserved; no new skill directories for this feature
**Scale/Scope**: 3 SKILL.md files, 3 docs files, 1 test file, 1 model config template; incremental rollout across 5 phases

## Exclusions

The following are **explicitly out of scope** for this feature and must not appear in implementation work:

- Persistent team or squad orchestration skills
- Fleet-style long-lived agent coordination
- New skill directories (e.g., `skills/team-orchestration/` or `skills/scout-discovery/`)
- Open-ended agent-to-agent conversation loops
- Multi-session shared memory or retained team context
- Production observability infrastructure (dashboards, telemetry pipelines)
- Expansion of existing Fleet or team-mode sections beyond clarifying that they remain optional and out of scope for this feature

These may be pursued as separate features that compose with the bounded workflow skills updated here. This feature only refines bounded coordinator behavior inside the three current skills.

## Research Findings

> Full analysis in [research.md](research.md).

### Decision 1: Scout/discovery as a skill phase, not a separate skill

- **Chosen**: Add a discovery phase to each existing skill's workflow section
- **Rationale**: Avoids new skill registration, manifest changes, test array updates, and plugin packaging changes. Keeps the three-skill split intact per CLAUDE.md conventions. The scout is a coordinator behavior (cheap model doing discovery before expensive delegation), not a standalone workflow protocol.
- **Alternatives considered**: (a) New `skills/scout-discovery/` skill — rejected because it breaks the three-skill invariant and requires plugin manifest, test, and packaging changes for a role that always operates within another skill's lifecycle. (b) Shared discovery library imported by all skills — rejected because skills are Markdown protocols, not code modules.

### Decision 2: Factual brief replaces "no shared context"

- **Chosen**: Replace "do not share context between roles" with "share factual brief, keep judgment separate" in skill text
- **Rationale**: The current wording at `parallel-implementation-loop/SKILL.md:48` is too blunt. It protects reviewer independence but forces expensive agents to rediscover the same files, validation commands, and task boundaries. The recommendations doc (line 46–51) demonstrates the distinction clearly: share facts (files, boundaries, commands), not conclusions (what the reviewer should think).
- **Alternatives considered**: (a) Keep current wording and rely on coordinator intelligence — rejected because the current wording actively instructs the coordinator to withhold useful context. (b) Share full context including conclusions — rejected because it undermines reviewer independence.

### Decision 3: Progress-based rescue as coordinator state model

- **Chosen**: Add a state model to the coordinator sections with soft budget → rescue → hard budget → stopped transitions
- **Rationale**: The current skills have stop conditions but no explicit middle ground between "keep waiting" and "stop." The recommendations doc (lines 139–166) provides a clean state diagram. Embedding this in the Workflow section of each skill gives coordinators explicit policy language.
- **Alternatives considered**: (a) Simple timeout with no rescue — rejected because it does not solve premature abandonment. (b) Unlimited patience — rejected because it creates runaway token spend. (c) External orchestration layer managing budgets — rejected as out of scope (persistent orchestration).

### Decision 4: Durable artifacts as execution requirements, not documentation guidance

- **Chosen**: Move artifact creation from "encouraged" to required in the Required Gates section of each skill
- **Rationale**: The existing artifact templates in `docs/workflow-artifact-templates.md` are good but advisory. Making them gate requirements means the coordinator must produce them before declaring work complete. This directly addresses FR-005 (durable artifacts) and SC-003 (every completed workflow produces artifact updates).
- **Artifact semantics**: "Durable" means a repository-appropriate sink that survives the active exchange. In repositories like this one, that may be a committed Markdown artifact under `docs/`. In other repositories, it may be a PR description, issue comment, task tracker entry, or another durable review surface. This feature requires durable handoff artifacts, not committed `docs/*.md` files in every repository.
- **Alternatives considered**: (a) Keep artifacts advisory — rejected because it does not satisfy FR-005's "MUST require" language. (b) Add tooling to enforce artifact creation — rejected because skills are Markdown protocols consumed by LLM runtimes, not code with runtime hooks.

### Decision 5: Convergence and disagreement rules as skill protocol additions

- **Chosen**: Add explicit convergence rules to each skill's Workflow and Stop Conditions sections
- **Rationale**: The spec (FR-006) requires convergence/disagreement rules. The recommendations doc (lines 196–206) provides concrete rules: same issue twice → escalate; scope growth → re-scope; material disagreement → higher-trust resolver or developer escalation. These belong in the skill protocol text where coordinators read them.
- **Alternatives considered**: (a) Global convergence document — rejected because rules need to be contextual per skill phase. (b) Coordinator-level meta-skill — rejected as persistent orchestration (out of scope).

### Decision 6: Outcome measures as artifact fields, not telemetry

- **Chosen**: Add outcome measure fields to the workflow artifact templates and require recording them in the batch/session summary artifact
- **Rationale**: The spec (FR-007) requires workflow-level outcome measures. Since skills run in LLM runtimes without telemetry infrastructure, the only durable recording mechanism is the artifact. The recommendations doc (lines 209–219) lists six metrics; the most actionable for skill-based recording are: discovery reuse (yes/no/skipped), rescue attempts, abandonment events, re-review loops, and final gate result.
- **Alternatives considered**: (a) Structured telemetry pipeline — rejected as production infrastructure outside scope. (b) No metrics at all — rejected because FR-007 requires them. (c) Session-only chat-based tracking — rejected because it does not survive sessions (violates FR-005).

## Data Model

> Full entity definitions in [data-model.md](data-model.md).

### Key Entities

| Entity | Purpose | Defined In |
|--------|---------|------------|
| Discovery Brief | Reusable factual summary for a bounded workflow execution | `docs/workflow-artifact-templates.md` (new template) |
| Workflow Track State | Durable record of one delegated unit of work with status transitions | `docs/workflow-artifact-templates.md` (new template) |
| Rescue Action | Bounded coordinator intervention for slow/stalled/mis-scoped work | Skill SKILL.md Workflow sections (inline policy) |
| Workflow Outcome Measure | Recorded indicator for throughput and quality assessment | `docs/workflow-artifact-templates.md` (new template section) |

### State Transitions

```
Track: pending → active → [review → revision →]* merged | abandoned | blocked
                active → rescue → active | serialized | abandoned
Rescue: re-scope | targeted-resend | serialize | escalate-to-developer
```
